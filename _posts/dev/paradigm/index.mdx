# 프로그래밍 패러다임

어떻게 이 코드를 이어나갈 것인가, 어떤 관점으로 코드를 짤 것인가를 선택하는데 결정적인 역할을 한다.
크게 명령형과 선언형으로 나눌 수 있다

## <span class="highlight">명령형(Imperative Language)</span>

: 문제를 어떻게 해결해야 하는지 컴퓨터에게 명령을 내리는 방법의 프로그래밍

> 알고리즘을 명시하고 목표는 명시하지 않는다

```c
const number = [1,2,3,4,5];
let sum = 0;

for (let i=0; i< number.length; i++){
  sum += number[i];
}
```

- 절차 지향 (ex: C)
- 객체 지향 (ex: C++, Java, Python)

### 절차 지향

실행 절차에 중점을 둔 프로그래밍 방식으로, 프로그램 전체가 유기적으로 연결되어 있다.

- **Top-Down 방식** : 하나의 큰 기능을 처리하기 위해서 작은 단위의 기능들로 나누어 처리
- **작은 규모**의 작업을 수행하는 함수 생성
- 인수, 반환 값으로 명령을 전달하고 수행
- 데이터와 함수를 별개 취급
- 특정 기능을 수행하려면 해당 메소드를 직접 호출

**장점**

- 개인 프로젝트에 적합
- 객체 지향에 비해 빠름

**단점**

- 대형 프로젝트에 부적합
- 유지보수가 어렵다
- 코드 순서가 중요. 순서 변경 시 다른 결과 초래 가능
- 분석과 디버깅이 어렵다

<br/>

### 객체 지향 (Object Oriented Programing)

필요한 데이터를 추상화시켜 상태와 행위를 가진 객체를 만들고,
그 객체들 간의 유기적인 상호작용을 통해 로직을 구성하는 프로그래밍 방법

기능별로 모듈화하여 같은 기능을 하드웨어가 중복하지 않게 하고, 모듈을 재활용하여 하드웨어의 처리량을 줄인다.

실행할 때마다, 값이 동일하다는 보장이 없다. property에 따라서 값이 달라진다.

- **Bottom-Up 방식** : 작은 문제를 해결하는 객체를 생성해 큰 문제를 해결

- **관계지향적** : class, property, method가 (객체의 관계를 중심으로) 한덩어리다.

- **Key Words**

  1. 클래스 + 인스턴스

  2. 추상화

     현실 세계의 대상을 관찰하여, 핵심적인 특징(속성, 행위)을 뽑아내는 과정
     우리가 구현하는 객체들이 가진 공통적인 데이터와 기능을 묶어 이름을 붙이는 것

     이때 관련 있는 것들을 묶으면 캡슐화! 묶은 개념은 추상화! 주체는 클래스!

  3. 캡슐화

     속성과 행위를 관련있는 것끼리 묶음

     관련된 데이터와 함수들을 클래스라는 하나의 '캡슐' 속에 그룹화

     접근이 필요한 부분을 제외하고 구체적인 로직은 내부로 숨긴다(정보은닉)

     - 접근 제한자: public, private, protected

  4. 상속

     대상이 되는 클래스의 모든 특징을 물려 받는 것

     - 계층형 구조
       - 상속관계 : 부모 클래스와 자식클래스
       - 관계가 아래로 내려갈수록 `구체화`(고유의 특징이 증가)
       - 관계가 위로 올라갈수록 `일반화`(더 많은 객체에게 영향을 준다)

  5. 다형성

     하나의 클래스 내부에 같은 이름의 기능을 여러개 정의하거나,
     상위 클래스의 기능을 하위 클래스에서 다시 정의할 수 있다.(오버로딩, 오버라이딩)

     - `오버라이딩`: 부모 클래스의 메서드와 같은 이름, 매개변수를 재정의
     - `오버로딩`: 같은 이름의 함수를 여러개 정의하고, 매개변수의 타입과 개수를 다르게 하여 매개변수에 따라 다르게 호출

즉, 객체 지향 프로그래밍은 실제로 우리가 사물을 바라보는 관점,
그리고 그 사물들의 연계성을 생각하는 관점을 프로그래밍에 적용하여 모델링 하는 패러다임이다.

<br/>

**장점**

- 코드 재사용이 용이
- 유지 보수가 쉬움
- 대형 프로젝트에 적합

**단점**

- 처리속도가 상대적으로 느림
- 객체가 많으면 용량이 커질 수 있음
- 설계시 많은 시간과 노력이 필요함

### 절차 지향 vs 객체 지향

절차 지향 => 데이터 중심으로 절차적 실행에 초점

객체 지향 => 객체간의 관계에 초점

##  <span class="highlight">선언형(Declarative Language)</span>

> 알고리즘을 명시하지 않고 목표만 명시한다.

```js
function add(x, y) {
  return x + y;
}
```

- 함수형 (ex: SQL, HTML, Python)

### 함수형

순수함수를 조합해 간결하게 만들고, side effect를 없앤다.

- 함수는 **순수함수**

  input에 대한 output이 1:1매칭.
  input에 대한 output이 보장됨.

- **Side Effect**

  순수함수는 side effect를 일으키지 않는다.

- **Immutability(불변성)**을 지향하기에 동작을 예측하기 쉽다.

  같은 입력에 항상 같은 값을 반환해야 한다.
  같은 input -> 같은 output
  즉, side effect가 없어야 한다.

- 루프보다는 **재귀**를 사용

  `for`, `while`과 같은 루프보다는 함수안에 함수를 호출하는 `재귀호출`을 사용하는 방법을 사용

  코드를 간결하게 하고 가독성을 높여 로직에 집중시킨다.
  람다식이나 Collection과 같은 API를 통해 가독성을 높힌다.

## 멀티 패러다임 언어

하나 이상의 프로그래밍 패러다임을 지원하는 프로그래밍 언어

> “하나 이상의 프로그래밍 스타일을 따르는 프로그램”을 허용한다. \_비야네 스트롭스트룹

모든 문제를 가장 쉽고 효율적으로 풀 수 있는 하나의 패러다임은 없다는 것을 인정하고,
프로그래머가 자신의 일에 가장 적합한 것을 사용할 수 있게 하는 것이다

- ex: JavaScript
